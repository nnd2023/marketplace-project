# Нефункциональные требования

Данный документ описывает нефункциональные требования (НФТ) к системе. Требования сгруппированы по атрибутам качества.

---

## 1. AVL. Требования к доступности (Availability)

| ID | Сценарий | Условия / Триггер | Ожидаемый результат | Критерий приемки |
| :--- | :--- | :--- | :--- | :--- |
| **AVL-001** | Доступность критических сервисов | Нормальная эксплуатация | Пользователи могут выполнять ключевые операции (вход, поиск, заказ, оплата) | Uptime ≥ 99.9% в месяц |
| **AVL-002** | Доступность второстепенных сервисов | Нормальная эксплуатация | Пользователи могут просматривать отзывы, рекомендации, бонусы | Uptime ≥ 99% в месяц |
| **AVL-003** | Плановое обслуживание | Окно тех. работ (ночное время) | Сервисы обновляются без полной остановки для пользователей | Возможность обновления без downtime или с < 30 сек простоя на сервис |

---

## 2. PRF. Требования к производительности (Performance)

| ID | Сценарий | Условия / Нагрузка | Ожидаемый результат | Критерий приемки |
| :--- | :--- | :--- | :--- | :--- |
| **PRF-001** | Поиск товаров в каталоге | 1000 RPS, 95-й перцентиль | Пользователь получает результаты поиска | Время ответа ≤ 200 мс (p95), ≤ 500 мс (p99) |
| **PRF-002** | Оформление заказа (Checkout) | Пиковая нагрузка, 500 одновременных чекаутов | Заказ успешно создается, оплата инициируется | End-to-end время ≤ 2 сек, без таймаутов и потерь |
| **PRF-003** | Загрузка карточки товара | Мобильное устройство, 4G-сеть | Страница полностью отрисована и интерактивна | LCP ≤ 2.5 сек, FID ≤ 100 мс, CLS ≤ 0.1 |
| **PRF-004** | Пиковая нагрузка (распродажа) | Увеличение трафика в 10× от базового | Система обрабатывает запросы без критических ошибок | Ошибки 5xx ≤ 0.1%, время ответа критических путей деградирует не более чем на 50% |
| **PRF-005** | Работа с кэшем | Высокая частота чтения каталога | Данные отдаются из кэша, нагрузка на БД снижена | Hit-rate Redis ≥ 80%, время ответа из кэша ≤ 10 мс |

---

## 3. REL. Требования к отказоустойчивости (Reliability / Fault Tolerance)

| ID | Сценарий | Условия / Сбой | Ожидаемый результат | Критерий приемки |
| :--- | :--- | :--- | :--- | :--- |
| **REL-001** | Отказ ноды базы данных | Падение primary-реплики БД | Система переключается на standby-реплику | Автоматический failover ≤ 30 сек, потеря данных = 0 (RPO=0) |
| **REL-002** | Отказ экземпляра сервиса | Падение пода/контейнера микросервиса | Запросы перенаправляются на здоровые экземпляры | Балансировщик исключает упавший инстанс за ≤ 10 сек, HPA запускает замену |
| **REL-003** | Потеря связи с внешним сервисом | Таймаут/ошибка платежного шлюза | Пользователь получает понятную ошибку, система не "зависает" | Применение Circuit Breaker: возврат fallback-ответа ≤ 3 сек, повторные попытки по экспоненциальной задержке |
| **REL-004** | Сбой брокера сообщений | Недоступность Kafka/RabbitMQ | Сообщения не теряются, обработка возобновляется после восстановления | Гарантированная доставка (at-least-once), идемпотентность потребителей, очередь персистентна |
| **REL-005** | Коррупция данных при сбое | Прерывание транзакции оплаты | Данные остаются в консистентном состоянии | Атомарность транзакций, механизмы саги/компенсирующих транзакций для распределенных операций |

---

## 4. OBS. Требования к мониторингу и наблюдаемости (Monitoring / Observability)

| ID | Сценарий | Условия / Событие | Ожидаемый результат | Критерий приемки |
| :--- | :--- | :--- | :--- | :--- |
| **OBS-001** | Сбор и агрегация логов | Любое событие в системе | Логи доступны для поиска и анализа в централизованном хранилище | Все сервисы пишут в stdout/structured JSON, сбор через Fluentd/Vector в ELK/PLG, хранение ≥ 30 дней |
| **OBS-002** | Метрики производительности | Непрерывная работа сервисов | Доступны метрики для анализа здоровья и производительности | Сбор метрик (Prometheus) с интервалом ≤ 15 сек: RPS, latency (p50/p95/p99), error rate, CPU, memory |
| **OBS-003** | Оповещение об инцидентах | Превышение порога ошибок или latency | Ответственная команда получает уведомление | Правила алертинга в Alertmanager, уведомление в PagerDuty/Slack ≤ 1 минуты после детекта |
| **OBS-004** | Аудит действий пользователей | Изменение критических данных (цена, статус заказа) | Действие зафиксировано в неизменяемом логе аудита | Запись в аудит-лог: кто, что, когда, до/после; лог защищен от модификации, хранение ≥ 1 год |
