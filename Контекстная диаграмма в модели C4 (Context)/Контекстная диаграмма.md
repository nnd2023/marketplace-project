## Обоснование выбора Внешних систем:

| № | Наменование Внешней системы | Описание функционала | Обоснование выбора |
| -- | ----------- | ---------------- | ---------------- |
| 1 | Платежный шлюз | Внешняя система обработки платежей (эквайринг). | Без наличия лицензии ЦБ и сертификации PCI DSS система не может обрабатывать банковские карты, электронные кошельки и другие платёжные методы. Вынос платежей на внешнего провайдера снижает архитектурную сложность, повышает безопасность и доверие пользователей.|
| 2 | Служба доставки | Внешняя система логистических операторов для расчета сроков и трекинга. | Логистика — это отдельная сложная отрасль. Интегрируясь с агрегаторами доставок (СДЭК, Почта, частные курьерские службы) через API, мы получаем актуальные тарифы и зоны покрытия без необходимости содержать свой штат логистов-расчетчиков. |
| 3 | Сервис уведомлений | SMS, Email и Push-шлюзы для оповещения пользователей. | Встроенная реализация отправки email и SMS усложняет архитектуру, требует настройки очередей, шаблонов, обработки отказов и защиты от спама. Использование готового облачного решения даёт: <br> - гарантию доставки <br> - единый канал для всех типов уведомлений<br> - управление шаблонами без развертывания кода<br> - аналитику |
| 4 | Банковская система / Налоговая | Система для выплат продавцам (трансферы) и фискализации чеков. |Прямые переводы физлицам и юрлицам требуют соблюдения законодательства (115-ФЗ в РФ, например). Специализированные сервисы выплат берут на себя проверку контрагентов и налоговое агентирование. Ручной ввод платежек в банк-клиент для тысяч продавцов невозможен. Нужен автоматический файл выгрузки или API для массовых транзакций.|
| 5 | CDN/Файловое хранилище | Хранилище медиа-контента (фото/видео товаров). | Карточки товаров содержат сотни тысяч изображений высокого качества. Хранить их на файловом сервере приложения нельзя (это замедлит работу, усложнит резервное копирование и масштабирование). Нужна объектная файловая система с глобальной сетью доставки контента (CDN) для быстрой загрузки картинок пользователю в любой точке мира. |


## Архитектура взаимодействий Cистемы

Ниже представлены протоколы и технологии взаимодействия, сгруппированные по архитектурным слоям.

### Группа А: Клиентские интерфейсы (Frontend & Mobile)
*Фокус: UX, производительность, работа при плохой связи.*

| Роль / Клиент | Основной протокол | Дополнительный протокол | Архитектурное обоснование |
| :--- | :--- | :--- | :--- |
| **Покупатель (Web)** | **REST / GraphQL** (HTTPS) | **WebSocket** (SSE) | REST для кэшируемого контента (каталог). WS только для live-статусов заказа и чата. Разделение снижает стоимость инфраструктуры. |
| **Покупатель (Mobile)** | **GraphQL** (HTTPS) | **Push Notifications** | GraphQL уменьшает количество сетевых запросов (важно для мобильного трафика). Push дешевле и энергоэффективнее постоянного WS соединения на телефоне. |
| **Продавец / Админ** | **REST** (HTTPS) | **WebSocket** | Тяжелые формы редактирования товаров требуют надежного REST. WS используется только для уведомлений о новых заказах или вопросах поддержки. |
| **Курьер (Mobile)** | **REST** (HTTPS) | **Local DB Sync** | Критичен режим **Offline-first**. Данные сохраняются локально и синхронизируются при появлении сети. Постоянное соединение не требуется и даже вредно в метро/подвалах. |
| **Склад (ТСД)** | **WebSocket** (TLS) | — | Единственный сценарий, где оправдан постоянный дуплексный канал. Требуется мгновенная реакция (<100мс) на сканирование штрих-кода на конвейере. |

### Группа Б: Внутренние инструменты (Back-office)
*Фокус: Целостность данных, аудит, аналитика.*

| Роль / Инструмент | Протокол | Обоснование выбора |
| :--- | :--- | :--- |
| **Поддержка (CRM)** | **REST** + **WebSocket** | REST для записи истории тикетов (аудит). WebSocket для отображения новых сообщений в реальном времени. |
| **Модератор** | **REST** + **WebSocket** | REST для фиксации решения (версионность контента). WebSocket для мгновенной выдачи следующей задачи из очереди. |
| **Менеджер (BI)** | **GraphQL** | Возможность гибко запрашивать именно те метрики, которые нужны для конкретного отчета, без создания множества эндпоинтов на бэкенде. |
| **Финансист** | **REST** (Idempotent) | Строгая гарантия доставки запроса. Механизм ключей идемпотентности предотвращает финансовые потери при сетевых сбоях. |
| **Сисадмин** | **Prometheus API** | Pull-модель сбора метрик является стандартом де-факто для микросервисных архитектур и Kubernetes. |

### Группа В: Внешние интеграции (Backend-to-Backend)
*Фокус: Надежность, безопасность, соответствие регламентам.*

| Внешняя система | Протокол / Паттерн | Направление | Обоснование |
| :--- | :--- | :--- | :--- |
| **Платежный шлюз** | REST + **Webhooks** | Двустороннее | Мы инициируем платеж (REST). Шлюз сообщает результат асинхронно (Webhook), чтобы не держать соединение открытым долго. |
| **Логистика (СДЭК/Почта)** | REST + **Webhooks** | Двустороннее | Аналогично платежам. Заказ курьера — синхронно. Обновление трека — асинхронно через webhook. |
| **Банк (Выплаты)**/**Налоговая (ФНС)** | **SFTP** (Reestry) + API/**Message Queue** + HTTP | Система → Банк/Система → Queue → ФНС | Массовые выплаты (payouts) регулируются строгими банковскими форматами файлов (XML/CSV). SFTP обеспечивает безопасную передачу реестров. Фискализация не должна блокировать оформление заказа пользователем. Отправка чека выносится в фоновый процесс через очередь. |
| **Сервис уведомлений** | **Message Queue** (Kafka/RabbitMQ) | Система → Queue | Асинхронная рассылка. Если сервис SMS упал, сообщения не теряются, а накапливаются в очереди до восстановления. |
| **CDN / Хранилище** | **S3 API** (Write) + **HTTP** (Read) | Разделенный поток | Прямая загрузка в облако (минуя наш сервер) и раздача контента через глобальную сеть CDN для максимальной скорости. |
